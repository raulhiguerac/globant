## Overview

REST API for CSV ingestion of historical hiring data into three tables: `departments`, `jobs`, `employees`. Domain inside the service: `ingestion`.

## Models (`src/app/models/models.py`)

- `IngestionBatch` — UUID PK + `created_at` (server_default). Tracks each ingestion request for audit and traceability.
- `IngestionBatchStatus` enum: `pending`, `completed`, `failed`. `failed` added via Alembic migration `1099b8b292cc` (`ALTER TYPE ingestion_batch_status ADD VALUE IF NOT EXISTS 'failed'` — Postgres enum values are not autogenerated by Alembic, this migration is hand-written).
- `AuditMixin` — `created_at` (server_default=func.now()) + `batch_id` FK → `ingestion_batches`. Applied to all three tables.
- `Departments`, `Jobs`, `Employees` — all columns NOT NULL. IDs come from CSV, not autoincrement.
- `Departments` and `Jobs` defined before `Employees` in the file — Alembic FK dependency order.
- `DateTime(timezone=True)` on all datetime columns.
- Column comments via `sa_column=Column(..., comment="...")`, table comments via `__table_args__` dict.

## Use Case Pattern

Applies to `IngestDepartmentsUseCase` and `IngestJobsUseCase` (employees has different flow — see below):

- `batch_ingestion_id = uuid.uuid4()` generated in UC, passed to both `batch.add` and `bulk_insert`.
- Happy path: `batch.add` + `bulk_insert` → single `commit()` — no orphan batch rows on failure.
- Fallback: row-by-row with savepoints. Commit only if `ok_count > 0`.
- Batch creation failure → `logger.error` + `raise` (critical, cannot continue without batch_id).
- Error translator (`translate_db_error` pattern) lives in the adapter, not the UC.

## IngestEmployeesUseCase

- `stream: BinaryIO` received in `execute()`, not `__init__` — it is operation input, not a dependency.
- Flow: upload stream to storage → `batch.add(pending)` → commit → return `batch_id`.
- Bulk insert + fallback row-by-row lives in the **worker**, not the UC (file can be large, cannot block request).
- Returns `batch_id` immediately; router enqueues `worker.stream_and_process` as `BackgroundTask`.

## Worker — `ProcessEmployeesChunkWorker`

The class itself is pure orchestration — no business logic. Business logic lives in `app/workers/helpers/`, each piece independently unit-tested:

- `storage_reader.download_file` — consumes the storage port's async chunk stream and returns the full CSV as `bytes`. Note: the GCS-level `stream_file` read is genuinely lazy (8KB reads via `blob.open("rb")`), but `download_file` buffers everything before returning — see [[metrics]] Claims and ADR note below for why this trade-off is accepted.
- `chunking.iter_chunks` — splits the parsed record list into chunks of `EMPLOYEE_CHUNK_SIZE`.
- `employee_writer.write_employees` — bulk insert one chunk + savepoint row-by-row fallback. Returns `list[str]` of failed rows.
- `batch_status.finalize_batch` — `update_status(completed)` + commit + `cache_client.delete(METRICS_CACHE_KEYS)`. Called once after all chunks.
- `batch_status.mark_batch_failed` — `update_status(failed)` + commit; swallows its own errors (logs only) so it never masks the original exception that triggered it.

Worker methods:

- `stream_and_process(batch_id)` — entry point scheduled via `BackgroundTasks`. Wraps `_process()` in a catch-all `try/except Exception`: on any failure (storage, DB, parse, or finalize) it calls `mark_batch_failed` and re-raises. This closes the "batch stuck in `pending` forever" gap for in-process crashes — see [[adr-0003-background-worker-session-and-failure-detection]].
- `_process(batch_id)` — re-streams file from storage (key `employees/{batch_id}.csv`), parses, chunks by `EMPLOYEE_CHUNK_SIZE`, loops `write_employees`, then `finalize_batch`.
- Must read from storage, not from the request file — the upload in the UC already consumed the stream.
- Cache keys invalidated: `hiring:metrics:hires_by_quarter`, `hiring:metrics:departments_above_mean`.

## GetBatchStatusUseCase

- `GET /ingest/employees/{batch_id}/status` — polling endpoint for the async employees flow.
- Reads `batch.status` as-is, **except**: if `status == pending` and `now - batch.created_at > settings.BATCH_TIMEOUT_SECONDS` (default 600s), the batch is treated — and persisted — as `failed` right there, no second poll needed.
- Covers the case the worker's own catch-all can't: the process being killed outright (OOM, container restart) before it ever reaches its `except` block. See [[adr-0003-background-worker-session-and-failure-detection]] for why a timeout heuristic was chosen over a real job queue.
- 600s is deliberately generous — batches are capped at 1000 rows (`batch_guard._MAX_ROWS`), so a real run (even with row-by-row fallback) finishes in seconds.

## Router (`api/routes/ingestion.py`)

- `POST /ingest/departments` + `POST /ingest/jobs` — sync, `200 IngestionResponse`.
- `POST /ingest/employees` — async, `202 AcceptedResponse(batch_id)` + `BackgroundTask`.
- Parsers live in `helpers/csv_parser.py`, not in the router.
- `_MAX_ROWS = 1000` constant in router (API contract, not runtime config).

## DI Wiring (`api/deps/`)

- `deps/shared.py` — `@lru_cache` singletons for cache and storage. Public getters have no `Depends` params to avoid the lru_cache + Depends incompatibility.
- `deps/ingestion.py` — per-request: `get_uow(session=Depends(get_session))` + UC providers for the three sync endpoints (departments, jobs, batch status).
- **Employees worker is the exception**: `get_process_employees_runner()` does *not* return a `Depends(get_uow)`-built worker. It returns the `run_process_employees_chunk` function reference itself. That function opens its own `Session(engine)` when it actually runs (inside the `BackgroundTask`, after the request has already returned). See [[adr-0003-background-worker-session-and-failure-detection]] for why — a `Depends`-provided session (or a worker built from one) is closed before `BackgroundTasks` run, regardless of how many DI layers wrap it.
- Route does `background_tasks.add_task(runner, batch_id=batch_id)` where `runner: Callable[..., Awaitable[None]] = Depends(get_process_employees_runner)` — keeps `dependency_overrides` usable in tests without leaking a request-scoped session into the task.

## DB Error Translation (`helpers/db_error_translator.py`)

- `OperationalError` → `IngestionDbUnavailableError` (503).
- `IntegrityError` + PK (`departments_pkey`, `jobs_pkey`, `employees_pkey`) → `RecordConflictError` (409).
- `IntegrityError` + FK (`employees_department_id_fkey`, `employees_job_id_fkey`) → `RecordForeignKeyError` (422).
- In bulk except: `except OperationalError` → translate + raise; `except Exception` → rollback + fall to row-by-row.
- In row-by-row except: `except OperationalError` → translate + raise; `except Exception` → rollback_to_savepoint + append to errors list.
- `IntegrityError` in bulk falls to row-by-row — allows rescuing valid rows from the same batch.

## Schemas

- `DepartmentRecord`, `JobRecord` — Pydantic models matching CSV columns exactly.
- `IngestionResponse(created_count: int, errors: list[str])` — shared across all 3 endpoints.
- Failed row format in errors list: `"id:name"`.

## Validation

- Max 1000 rows per request per endpoint.
- Parser dep validates `len(records) <= 1000` and returns 400 if exceeded.

## Infrastructure

- `pyrightconfig.json` at service root with `extraPaths: ["src"]` — required for Pylance to resolve `app.*` imports from `src/app/`.

## Claims

- `IngestionBatch` must be persisted in the same transaction as the records — FK constraint enforces this.
- `server_default=func.now()` is used (not Python-side `default_factory`) for all `created_at` columns.
- IDs on all three tables are provided by CSV — no autoincrement.
- Error translator lives in `services/ingestion/helpers/`, not in the UC or adapter.
- Max batch size is 1000 rows per request.
- `stream_and_process` reads the file from storage, not from the original request object.
- `@lru_cache` singletons must not receive `Depends()` params — call the cached function directly inside the getter.
- `SqlUnitOfWork.__init__` initializes `self._savepoint = None` — without it, calling `rollback_to_savepoint()` before any `begin_nested()` raises `AttributeError` (fixed 2026-07-05; was a latent bug, not yet triggered by any current flow).
- `download_file` (in `workers/helpers/storage_reader.py`) buffers the entire CSV into memory before `parse_employees` runs — `stream_and_process`'s name promises more than it does at the parsing layer. Accepted trade-off for the ~2MB challenge CSV; avoids handling records split across chunk boundaries.
